3. Complete System Flow — Step by Step 
 
Step 1 — PHP Backend Sends Request 
The OpenCart PHP backend collects the logged-in user's recent purchase history and sends it 
as a POST request to the Flask server running on port 5000. 
 
Step 2 — Content-Based Filter (Candidate Generation) 
This is the core of the candidate generation phase. The system analyzes the user's purchase 
history to understand their preferences: 
 
• Extracts the subcategories from the user's history (e.g., Hoodies, Sweatpants) 
• Identifies the top 3 most purchased subcategories 
• Generates 50 candidates: 70% from preferred subcategories + 30% from other 
subcategories for diversity 
• Excludes products already purchased by the user 
 
Example: User bought Hoodies + Sweatpants + T-Shirts 
  → 35 candidates from Hoodies, Sweatpants, T-Shirts (70%) 
  → 15 candidates from Jackets, Pants, Accessories (30% diversity) 
 
Step 3 — LightGBM Ranks the 50 Candidates 
The LightGBM ranker takes the 50 candidates and scores each one using real features from the 
product database and the user's purchase profile: 
 
Feature Source Description 
category_encoded products_clean.csv Encoded main category of the candidate product 
subcategory_encoded products_clean.csv Encoded subcategory of the candidate product 
price products_clean.csv Actual price of the candidate product 
price_normalized products_clean.csv Price scaled between 0 and 1 
fav_category user_profiles.pkl The user's most purchased category 
fav_subcategory user_profiles.pkl The user's most purchased subcategory 
avg_price user_profiles.pkl The user's average spending per product 
purchase_count user_profiles.pkl Total number of purchases by the user 
 
Step 4 — Diversity Filter 
After LightGBM ranks all 50 candidates, a diversity filter ensures the Top 5 results are not all 
from the same subcategory. Maximum 2 products from any single subcategory are allowed in 
the final results. 
 
Step 5 — Top 5 Results Returned to PHP 
The Flask API returns the top 5 ranked products as a JSON response:

5. Content-Based Filter — What It Does Exactly 
The Content-Based Filter replaces the GRU model as the candidate generator. It works by 
analyzing what the user has already purchased and finding similar products: 
 
How It Works 
1. Extract subcategories from the user's purchase history 
2. Count frequency of each subcategory (e.g., Hoodies: 5 times, T-Shirts: 3 times) 
3. Select top 3 most purchased subcategories as 'primary preferences' 
4. Generate candidate pool: 70% from primary + 30% from other subcategories 
5. Remove already-purchased products from candidates 
 
Why This Is Better Than Random Selection 
• Ensures candidates are relevant to the user's actual taste 
• The 30% diversity prevents the system from only recommending the same subcategory 
• Excludes already-purchased products to avoid repetition 
• Works for both existing users and cold-start users 
 
 
6. LightGBM Ranker — What It Does Exactly 
LightGBM (Light Gradient Boosting Machine) is the core ranking model. It takes the 50 
candidates from the Content Filter and assigns a relevance score to each one. 
 
How It Works 
• For each of the 50 candidates, it creates a feature vector combining product features + 
user profile features 
• It predicts the probability that the user will purchase each candidate (score between 0 
and 1) 
• The higher the score, the more relevant the product is for this specific user 
• Candidates are ranked by score and the Top 5 are returned 
 
Why LightGBM? 
Feature Benefit 
Works with tabular data Product features and user profiles are structured data — perfect for 
LightGBM 
Fast inference Ranks 50 products in milliseconds 
Handles imbalanced data Uses class_weight='balanced' to handle the fact that users buy few 
products out of many 
Interpretable Can explain why a product was recommended using feature 
importance
